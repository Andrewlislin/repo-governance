# Signed release catalog

The version advisory trusts neither HTTPS alone nor JSON shape alone. `release-catalog.json` conforms to `schemas/release-catalog.schema.json` and uses a canonical compact JSON representation with one trailing line feed. Releases are ordered by strictly increasing SemVer and timestamp; version and full commit SHA values are unique. `release-catalog.sig` is a base64-encoded Ed25519 detached signature over those exact catalog bytes.

The canonical download endpoints are fixed to the latest release of `CoaseEdge/repo-governance`. The client accepts only HTTPS, validates the repository path on `github.com`, validates the immutable GitHub repository asset ID, release-asset path, and filename on `release-assets.githubusercontent.com`, and repeats validation at every redirect. The executable embeds one DER/SPKI Ed25519 public key. Key rotation, revocation, and an offline root-key design are intentionally outside this version; changing that key requires an explicit source and executable release.

The tracked `release-catalog.json` is the historical input. `release-metadata.json` declares whether the next source release is a security fix. During a tag release, `scripts/write-release-catalog.mjs` appends the package version, source commit, deterministic commit timestamp, and security flag, then signs with `REPO_GOVERNANCE_CATALOG_PRIVATE_KEY`. That private key is available only as a GitHub Actions secret and must never be committed, printed, copied into an artifact, or supplied to ordinary CI. `scripts/verify-release-catalog.mjs` verifies the detached signature with the same public key embedded in the CLI and checks that the final record matches the release being published. Both files are uploaded with the immutable release assets.

`repo-governance version check` explicitly performs the network refresh. A verified catalog and signature are stored together in one atomic cache envelope under the standard user data directory. A failed refresh may use an older, still-valid cache and reports `cached`; a lower catalog version is rejected as rollback. A damaged cache is never used.

`preflight` only reads and verifies that cache. It performs no network request and no write, including no “last warned” timestamp. Its deterministic `updateAdvisory` reports availability, current and latest versions, number of published releases behind, whether a security fix is available, whether a warning should be shown, a reason, and catalog status. The pre-push path continues to run `check` and never evaluates or prints this advisory.
