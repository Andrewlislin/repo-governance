# repo-governance v1.1.1 Agent preflight plan

Version 1.1.1 adds a read-only `repo-governance preflight [--json]` command as the single governance entry point before an Agent starts repository work. The CLI remains the only policy engine. Agent Skills, Claude commands, and optional lifecycle hooks consume the stable JSON report and translate it into platform behavior; they never reimplement RG001-RG005, preset selection, path matching, or engine identity checks.

The supported workflow is:

```text
Agent receives repository work
  -> preflight --json
  -> CLI classifies governance state and authorization policy
  -> Agent continues, asks for confirmation, or stops
  -> first adoption uses bootstrap/new/clone
  -> later pushes use pre-push
  -> pull request preparation uses prepare-pr
```

The Agent gate and optional hook templates are explicitly installed and trusted. Version 1.1.1 does not introduce system-level interception, silent bootstrap, or a plugin hard gate.

## Public preflight contract

`preflight` is offline and read-only. It does not modify a repository, Git hook, user configuration, or GitHub state, and it does not execute RG001-RG005. It inspects the canonical Git root, governance configuration and schema, locked engine alignment, and whether the current repository has an effective pre-push connection to the stable dispatcher.

The stable report retains `schemaVersion: 1` and adds these required semantics:

- `ok` states whether inspection and classification completed. It does not authorize repository writes.
- `status` states whether governance permits the Agent to continue. Adapters decide from `status`, `repoState`, and `recommendedAction` together.
- `exitCode` mirrors CLI compatibility semantics: `0` for success, `1` when governance needs attention, and `2` for invocation, environment, policy, or safety blocking.
- `cwd` is always the normalized absolute invocation directory.
- `repoPath` is the normalized absolute Git root, or `null` outside a Git repository.

The stable repository states and legal outcomes are:

| `repoState` | `ok` | `status` | `exitCode` | Meaning |
| --- | --- | --- | --- | --- |
| `managed` | `true` | `succeeded` | `0` | Configuration, locked engine, and effective hook are valid. |
| `not_git_repo` | `true` | `needs_attention` | `1` | `cwd` is valid, `repoPath` is `null`, and repository writes must not start. |
| `unmanaged` | `true` | `needs_attention` | `1` | The Git repository has no governance configuration. |
| `misconfigured` | `true` | `needs_attention` | `1` | Configuration, engine alignment, or effective hook connection needs repair. |
| `blocked` | `false` | `blocked` | `2` | Inspection cannot complete safely or policy input is invalid. |

In particular, `unmanaged` legitimately reports `ok: true` while requiring attention. Adapters must never use `ok` alone as permission to write.

The `inspection` object always exposes `gitRepository`, `configPresent`, `configValid`, `engineAligned`, and `hookConnected`. Facts that cannot apply or cannot be verified in a source development identity use `null`; they are never presented as verified. A disconnected hook keeps `configValid: true` and `engineAligned: true`, sets `hookConnected: false`, classifies the repository as `misconfigured`, and recommends `hook-reconnect-required`. Read-only diagnosis may continue, but edits and commits stop until the hook is restored.

`recommendedAction` always contains `id`, `requiresPreset`, and `requiresConfirmation`. Misconfigured reports preserve the original stable error code and CLI diagnosis instead of encouraging an Agent-side configuration repair.

## User authorization policy

The optional user file is `~/.repo-governance-agent.json`:

```json
{
  "schemaVersion": 1,
  "autoPreflight": true,
  "autoBootstrap": false,
  "defaultPresetByPath": [
    {
      "pathPrefix": "/Users/example/Projects/web",
      "preset": "react-web"
    }
  ]
}
```

Missing policy uses built-in defaults: `autoPreflight: true` and `autoBootstrap: false`. Invalid policy fails closed. `autoBootstrap: true` is legal only with `autoPreflight: true` and an explicitly matched preset. Matching uses normalized real paths, chooses the longest path prefix, and blocks equal-priority conflicts. Presets pass through the existing preset loader and schema.

The policy does not allow an Agent to infer a preset. It only authorizes an explicit `bootstrap` call with an already selected preset. It never authorizes `github enforce --confirm`, pull request creation, comments, ruleset changes, or other remote writes. Path policy applies only to an existing Git repository in version 1.1.1; `new` and `clone` still ask when no preset was explicitly supplied.

## Coding changes

Each Coding change implements exactly one `feature-list.json` node. After this initializer, only the current node's `passes` field may change.

1. `v1.1.1-pr-1-preflight`: implement the classifier, CLI help and human summary, stable report schema conditions, shared locked-engine inspection, strict effective-hook diagnosis, zero-write proofs, and state/exit regression tests.
2. `v1.1.1-pr-2-codex-gate`: add the shared `repo-governance-agent-gate` Playbook, Codex Skill, explicit opt-in `SessionStart` and `PreToolUse` templates, hook runner, adapter contract fields, fixture, and installation layout.
3. `v1.1.1-pr-3-claude-gate`: add the matching Claude command and explicit opt-in hooks, update `CLAUDE.md`, and require both adapters to consume the same decision fields and fixture.
4. `v1.1.1-pr-4-agent-policy`: implement policy schema, loading, real-path normalization, deterministic longest-prefix matching, conflict blocking, preset validation, and the remote-write authorization boundary.
5. `v1.1.1-pr-5-release`: update bilingual documentation and adapter guides, add complete state examples, include new assets in release digests, update version and manifests to 1.1.1, and run build/package/install acceptance.

## Verification

All tests use Node.js 22, temporary homes, temporary Git repositories, and offline fixtures. No development or test path enrolls an existing repository.

PR1 covers non-Git, unmanaged, managed, invalid configuration, engine mismatch, missing engine, disconnected hook, unreadable input, conditional schema rules, stable error preservation, and byte/permission snapshots proving zero writes. Adapter acceptance covers unmanaged confirmation and bootstrap, direct managed continuation, non-Git refusal, disconnected-hook read-only diagnosis, explicit and missing preset behavior, and `prepare-pr` before a pull request.

Cross-adapter tests require the same `schemaVersion`, `repoState`, `recommendedAction`, decision fields, and classification labels. Static scans reject wrapper implementations of RG rules, preset inference, policy matching, and remote-write authorization. Each change runs its fastest focused test first, followed by `npm run check:static` and `npm test`. Final release acceptance also runs the Node 22 SEA build, `npm pack`, release/install smoke tests, and the repository's own governance check.

## Fixed boundaries

- The release version is 1.1.1 and automation reports remain schema version 1.
- New commands and fields are backward-compatible additions; existing command semantics do not change.
- Native `git clone` and `git init` are not intercepted.
- No preset is guessed, no bootstrap is silent, and no GitHub write is implicit.
- Agent preflight discovers and explains governance state; Git pre-push remains the push defense; `prepare-pr` remains the pull request readiness check.
